## 記事の型

以下の流れで書く。

```
① 現場あるある（共感）
   → 読者が「これ自分の職場の話だ」と感じる書き出し

② 課題の深掘り
   → 品質への影響・属人化・原因特定の難しさなど、現場を知っているからこそ書けるコンテキスト

③ アイデアの核心
   → シンプルな原理をフローで説明。難しいアルゴリズムは不要という安心感を伝える

④ 実装
   → コードは3ブロック構成（読み込み・処理・出力）でシンプルに
   → サンプル画像はリポジトリに同梱して手元で動かせるようにする

⑤ 現場に適用するときのポイント
   → キャリブレーション・固定・ログ保存など、実運用に必要な視点を添える

⑥ 実装上の課題と対応アイデア
   → 導入前に想定すべき課題を「課題→対応アイデア」のセットで書く
   → 「まず試すこと→それでもダメなら→根本対策」の順で段階を示す
```

## コードの規約

**共通設定**
- Python 3.9+
- 依存パッケージ：`opencv-python` / `numpy` / `matplotlib`
- パラメータは冒頭にまとめてコメントを丁寧に書く

**サンプル画像の命名規則**
- 基準画像：`reference.png`
- 検査画像：正解値をファイル名に含める（例：`test_dx+3.0_dy+0.0.png`）
- 現場適用時の検査画像：ワークID＋撮影時刻（例：`W-00123_20240315_143022.png`）

**ディレクトリ構成（各コードリポジトリ）**
```
project_name/
├── main.py                 # メインコード
├── generate_samples.py     # サンプル画像生成（動作確認・追加用）
├── requirements.txt
├── .gitignore
├── README.md
└── samples/                # サンプル画像（リポジトリに同梱）
```

Zenn記事URL形式：`https://zenn.dev/factory_dx_eng/articles/スラッグ名`

## 過去の壁打ちから得た知見

**記事コンセプトについて**
- 業種（ガラスライン等）は記事に出さない。「搬送ラインのある製造業なら業種問わず使える」という書き方が読者層を広げる
- 会社のコードは使わない。公開データセットや自作サンプル画像で再現する
- Zennの方針：著者自身の経験・検証・洞察が入っていれば、AIと協働して作った記事でも問題ない

**ストック収益への展開**
- 記事3〜5本 → Zenn Book（有料）→ Boothでテンプレート販売
- 記事は「使いに来る場所」にする：ZennはSEO、GitHubは信頼性の証明

**文体について**
- 「現場を知っている人間が書いている」感を出す
- 接続詞・語尾が単調にならないよう注意
- まとめは表だけで終わらず、記事全体の主張を締める一文を添える
